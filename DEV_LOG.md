# DEV_LOG — yt-sub-md

开发日志：记录项目起源、迭代决策、踩坑与方案选择。

---

## 1. 项目起源

**原始需求来源：** 用户日常工作流中大量使用 Obsidian 管理知识，需要把 YouTube 上的教程、演讲、纪录片字幕批量导入笔记库。

**核心痛点：**
1. 手动复制字幕太繁琐，尤其带时间轴的字幕需要逐条删除时间戳
2. 很多视频默认只显示翻译字幕，但用户更想学习原语言内容
3. 浏览器里打开十几个视频想批量处理，没有现成工具支持
4. 市面上大多数字幕下载器要么输出带时间轴的 SRT/JSON，要么 CLI 体验粗糙

**项目目标：** 做一个"贴链接、回车、出 Markdown"的极简工具，零配置即可融入 Obsidian 工作流。

## 3. 踩坑记录

| 问题 | 根因 | 解决方案 | 涉及版本 |
|------|------|---------|---------|
| `type object 'YouTubeTranscriptApi' has no attribute 'list_transcripts'` | `youtube-transcript-api` 从 v0.x 升级到 v1.x 后，类静态方法全部改为实例方法 | 所有调用点改为 `YouTubeTranscriptApi().list()` 和 `YouTubeTranscriptApi().fetch().to_raw_data()` | v0.1.1 |
| 粘贴三个链接只识别到一个 | `typer` 的 `Prompt.ask()` 底层调用 `input()`，只读取一行输入 | 参考 `github-repo-downloader` 的设计，改用 `while True: line = input(" > ")`，空行结束，合并后统一 split | v0.1.2 |
| 英文视频下载到了中文字幕 | `_resolve_language` 策略是"第一个手动字幕优先"，不区分原语言和翻译；很多英文视频上传者同时提供了 `zh` 手动字幕 | 引入 `meta.original_language`（yt-dlp 提供），优先匹配该语言的手动/自动生成字幕 | v0.1.3 |
| 开头获取 73 个视频元数据卡死 2 分钟 | 元数据获取是串行同步调用 yt-dlp，每个 1~3 秒 | 改为每个下载 task 内部自己获取元数据（`asyncio.to_thread(fetch_metadata)`），并发执行 | v0.1.3 |
| `download_one` 阻塞整个事件循环 | `download_one` 是同步函数，内部调用 `YouTubeTranscriptApi().list()` 和 `fetch()` 会阻塞 asyncio 事件循环 | 所有同步 IO 调用都包在 `asyncio.to_thread()` 中执行 | v0.1.3 |
| Windows 文件名含保留字符报错 | YouTube 视频标题中常包含 `:`, `?`, `"` 等字符 | 在 `config.py` 中定义 `FILENAME_BAD_CHARS`，下载前统一替换为 `_` | v0.1.0 |
| `fetch()` 返回的不是 `list[dict]` | v1.x 中 `fetch()` 返回 `FetchedTranscript` 对象（dataclass），字段结构相同但类型不同 | 调用 `.to_raw_data()` 获取兼容旧代码的 `list[dict]` | v0.1.1 |
| Windows 控制台 `console.print()` 报 `UnicodeEncodeError` | Windows 默认控制台编码为 GBK，Rich 的某些符号（如 `ℹ`、`⚠`）无法编码 | 在 `_collect_tasks` 中封装 `_log()` 辅助函数，`try/except` 捕获异常后降级为普通 `print()` 输出 | v0.1.4 |

## 4. 设计决策

### 决策一：为什么用 yt-dlp + youtube-transcript-api 双库，而不是只用其中一个？

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 只用 yt-dlp | 一个库搞定所有，能下载字幕也能下载视频 | 字幕获取不是 yt-dlp 的核心能力，解析字幕列表的逻辑较粗糙，且依赖 ffmpeg 等外部工具时配置复杂 | ❌ 不选 |
| 只用 youtube-transcript-api | 轻量、专门做字幕、API 简洁 | 无法获取视频标题、频道、时长、播放列表结构 | ❌ 不选 |
| **yt-dlp（元数据）+ youtube-transcript-api（字幕）** | yt-dlp 负责它擅长的元数据和播放列表展开；youtube-transcript-api 专门负责字幕获取，两者互补 | 多一个依赖，但安装都是 `pip install` 一键完成 | ✅ **选中** |

### 决策二：为什么默认"原语言优先"，而不是默认英语或让用户每次指定？

**背景：** YouTube 视频的字幕通常有几种：上传者手动打的手写字幕、YouTube 自动生成的 ASR 字幕、以及从其他语言翻译过来的字幕。

**推理：**
1. 用户的核心使用场景是"学习原内容"，翻译字幕质量参差不齐，常出现漏翻、错翻
2. 如果默认英文，对中文/日文/德文等非英语视频用户不友好
3. 如果每次都要手动指定语言，批量处理时体验极差

**策略设计：**
1. 用户指定 `--lang` → 强制下载该语言
2. 无指定 → 优先手动字幕（上传者提供的，通常就是原语言）
3. 无手动字幕 → fallback 到第一个自动生成的字幕（通常也是原语言）
4. 都没有 → 最后才尝试翻译字幕

### 决策三：为什么去掉时间轴？

**反对意见：** 时间轴有助于精确定位某句话在视频中的位置。

**支持意见：**
1. 用户目标是"阅读内容"而非"精确定位"，时间轴是纯噪音
2. 去掉时间轴后，Markdown 可以按语义重新分段，阅读体验接近一篇长文章
3. Obsidian 的全文搜索和双向链接功能在纯文本上工作得更好，带时间轴的文本会干扰搜索相关性
4. 如果需要定位，Markdown 头部的 `**链接:**` 字段可以直接跳转到视频

**结论：** 输出为无时间轴的 Markdown，语义分段。

### 决策四：为什么用 asyncio + Semaphore 做并发，而不是线程池？

| 方案 | 分析 | 结论 |
|------|------|------|
| 串行下载 | 简单，但批量时极慢（20 个视频 × 5 秒 = 100 秒） | ❌ 太慢 |
| threading.ThreadPoolExecutor | IO 密集型场景下，GIL 不是瓶颈，但线程开销较大 | 可行，但不是最优 |
| **asyncio + Semaphore + asyncio.to_thread** | 轻量、适合 IO 密集型；`to_thread` 把同步库（yt-dlp, youtube-transcript-api）的阻塞调用丢到线程池，不阻塞事件循环 | ✅ **选中** |

### 决策五：为什么元数据获取要"随下载随获取"，而不是先全部获取完再下载？

**背景：** 73 个视频的批量测试中，串行获取元数据花了约 2 分钟，用户在这 2 分钟内完全看不到任何进度反馈，体验极差。

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 先全部获取元数据，再批量下载 | 逻辑简单，下载阶段知道总进度 | 开头卡死，大批量时用户体验差 | ❌ 不选 |
| **每个 task 自己获取元数据 + 下载** | 第一个视频几乎立刻开始处理，用户立刻看到反馈；失败一个不影响其他 | 需要在每个 task 内部处理元数据异常 | ✅ **选中** |

### 决策六：双语模式为什么用"逐句对齐"而不是"先英文全文再中文全文"？

**背景：** 用户要求"每句英文的下面一行加上中文的"。

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 英文全文放前面，中文全文放后面 | 适合只读英文或只读中文的场景 | 对照阅读时需要上下翻页/滚动，体验差 | ❌ 不选 |
| **逐句交错（英文+中文）** | 一眼就能看到对应翻译，精读体验最好；Obsidian 中搜索时上下文完整 | 行数翻倍，单页内容更少 | ✅ **选中** |

### 决策六：为什么播放列表处理要"自动识别"而不是单独命令/参数？

**背景：** 用户反馈"我提供链接的时候，可能是和视频链接一起提供的，换行分隔"。

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 单独 `--playlist` 参数 / `playlist` 交互选项 | 意图明确，代码分支清晰 | 用户需要事先区分哪些是播放列表、哪些是单视频，分批处理；混合链接时需手动拆分 | ❌ 不选 |
| **URL 自动识别（`is_playlist_url`）** | 用户只管粘贴，零心智负担；视频和播放列表可以任意混合；代码在 `_collect_tasks` 中统一处理 | 需要为每个 URL 做一次类型判断，开销极小 | ✅ **选中** |

**实现细节：**
1. `core/extractor.py` 中已有 `is_playlist_url()` 函数，判断 `"playlist?list=" in url`
2. `_collect_tasks()` 遍历所有链接，播放列表链接调用 `_expand_playlist()`（yt-dlp 展开），生成 `DownloadTask(url=video_url, output_dir=playlist_subfolder)`
3. 单视频链接生成 `DownloadTask(url=video_url, output_dir=base_output)`
4. 去重键为 `(video_id, str(output_dir))`，确保同一视频在不同目录下可共存
